# 工程经验库学习指南

> **本文档的定位**：把"怎么用这个经验库学一个开源项目"的方法固化下来。它不是经验卡，不进 `知识库/经验/`；它是 `工程经验库/` 的元文档，和 `开源项目工程经验提炼提示词模板.md`、`{项目名}-代码地图.md` 并列。
>
> **读者**：几个月后的你自己，或第一次打开这个目录的人。

---

## 0. 先读懂这个目录有什么

```
工程经验库/
├── 开源项目工程经验提炼提示词模板.md   ← 方法论（13 个场景 + 4 条纪律）
├── {项目名}-代码地图.md               ← 场景一的产出（项目总纲）
├── 学习指南.md                        ← 本文档（怎么用上面两样学一个项目）
└── 知识库/
    └── 经验/
        ├── 总目录.md                  ← 已入库经验的导航
        ├── 架构/                      ← ADR（架构层）
        └── 方案/                      ← 方案卡 + 原子卡（方案层 / 原子层）
```

**三样东西的分工**：

| 文档 | 回答什么 | 何时读 |
|---|---|---|
| 提示词模板 | "提炼经验时该用什么 prompt、按什么纪律" | 第一次建库 / 提炼新经验前 |
| 代码地图 | "这个项目的骨架是什么、P0 该读哪几个模块" | 接触一个新项目时 / 决定下一步读什么时 |
| 学习指南（本文） | "我怎么用上面两样真正学到东西" | 每次开始一轮学习前 |

如果你是第一次来，按 `学习指南 → 代码地图 → 总目录` 的顺序读一遍，再决定从哪个模块切入。

## 1. 学习的心智模型：经验库是"待验证的假设集"，不是"读完就会的书"

**最常见的误区**：把经验卡当成"知识已经装进脑子了"的凭证——读完一张卡就以为自己会了。

**真相**：经验卡是**别人（或过去的你）从源码里提炼出来的假设**。它的价值在你做两件事时才兑现：

1. **反向核验**：打开卡里附录指向的源码行号，自己看一遍，确认卡里写的和代码一致
2. **正向使用**：在自己项目里用这条经验时，发现卡的约束条件、代码骨架是否真的够用

没做这两步，经验卡只是"看着像知识的文字"。这就是为什么前面对 6 条经验做核对时能挑出行号偏差、函数名错、语义反了——**那些错误如果不核验，会一直藏在库里**。

## 2. 三阶段学习路径

### 阶段 1：验活已有经验（1-2 天）

库里有 6 条 reviewed 经验，但**读过 ≠ 吃透**。做法是逐条反向核验——就是模板场景五的审查动作，但这次审查者是"想学会"的你，不是"想入库"的你。

**核验清单**（每条经验都要走一遍）：

| 经验 | 核验动作 | 学到的点 |
|---|---|---|
| 统一操作契约 | 打开 `operations.ts` 的 `Operation` 接口，把 7 个字段抄一遍，问自己"为什么 `scope` 可选而 `remote` 必填" | trust 边界比权限分级更底层 |
| 双轴路由 | 找一个 read op（如 `get_page`），追它怎么调 `sourceScopeOpts` | 看一处真实 spread，理解"漏一处即泄漏" |
| 信任边界 | 在 `dispatch.ts` 把 `buildOperationContext` 的默认值逻辑默写一遍 | `remote ?? true` 比 `remote: false` 默认更安全 |
| 读侧作用域原子卡 | 自己造 5 个测试用例（卡里"最小验证清单"），在脑中跑一遍 | federated `[]` 不等于"读全库"这条最容易踩 |
| KnobsHash 方案卡 | 打开 `mode.ts` 的 `knobsHash` 函数体，把 parts 数组**完整**抄一遍，数清楚 33 个字段 | append-only 纪律为什么是结构性的 |
| KnobsHash 原子卡 | 问自己：如果我要加一个新 knob `foo`，改哪 3 处？ | parts + KNOBS_HASH_VERSION + 测试 |

**这一步的关键纪律**：不是"相信卡里写的"，而是"卡里写的每一条，我都打开源码看一眼"。如果发现卡和代码对不上——**先修卡，再学**。这是经验库不变成水文的根本保障。

走完这一步，6 条经验才真正变成你的。

### 阶段 2：补全 P0 未提炼的模块

代码地图第 10 节列了 6 个 P0 子任务，总目录的"下一步计划"也对应这 6 条。按模板的固定流程走每一条：

```
场景二（模块深读）→ 场景三（流程深读）→ 场景 4C（原子卡）→ 场景五（审查）→ 场景六（入库）
```

**建议的切入顺序**（按阻力从小到大）：

1. **hybrid search 全链路** —— 你已经有 KnobsHash 基础，顺势深入阻力最小
2. **sync + import-file + embed-backfill** —— lock + checkpoint + pace 是经典的可恢复系统模式
3. **cycle/dream** —— protected phase + spend gate，状态机味道重
4. **minions/jobs** —— 后台作业调度，与 sync 的 pace mode 有关联
5. **operations + dispatch** —— 信任边界的完整 enforcement 清单
6. **schema-pack + extract** —— 页面类型 taxonomy，偏内容侧

**每条要做的事**：

- 场景二模块深读：产出一份模块深读笔记（不入库，是中间产物）
- 场景三流程深读：产出一份流程深读笔记 + Mermaid 图
- 场景 4C 原子卡：从流程里切出**一个**阶段，写一张原子卡
- 场景五审查：用模板的硬性门槛 A-F 过一遍，不通过就打回重读
- 场景六入库：整理成最终 Markdown，更新总目录

**两个反模式提醒**：

- **不要跳过场景二直接做场景三**。模块深读是建立"对象地图"，流程深读才有"调用链"的锚点。跳过模块深读，流程深读会变成"看一行忘一行"。
- **觉得卡"空"就往 4C 切**。停在 4B 方案卡描述"用 RRF 融合"是空的；切到 4C 原子卡落到"score = 1/(60+rank)，A/B 召回各自从 rank 1 起算，加权后降序"才有干货。模板第 4C 节自己说：觉得 AI 还是空，多半是停在 4B 没往 4C 切。

### 阶段 3：跨项目归纳（场景十二）

gbrain 读到一半（建议阶段 2 完成 2-3 个模块后）就可以开始了。配读 1-2 个同类项目做对照：

| 对照维度 | 建议项目 | 对照 gbrain 哪条经验 |
|---|---|---|
| 检索缓存 | lancedb、chroma 的 hybrid search | KnobsHash 缓存指纹 |
| Agent 工具 | langchain tools、letta tool registry | 统一操作契约 |
| 多租户隔离 | supabase RLS 设计 | 双轴路由、信任边界 |
| 可恢复导入 | git-lfs、docker layer store 的 resumable fetch | sync checkpoint + lock |

读完后跑场景十二（跨项目归纳），产出"强经验"——多个项目验证过的设计。这是经验库最有价值的部分，单项目提炼不出这种层次。

**纪律**（模板场景十二明确要求）：不要因为多个项目都用了某设计就默认它普适。必须说明适用约束与迁移成本。

## 3. 模板场景到学习环节的映射

模板有 13 个场景，但学习时不是按编号顺序用，而是按"当前在学什么"选用：

| 你正在做 | 用模板哪个场景 |
|---|---|
| 第一次接触一个新项目 | 场景一（项目总纲）→ 产出代码地图 |
| 想搞懂某个模块的内部结构 | 场景二（模块深读） |
| 想搞懂一条端到端流程 | 场景三（流程深读） |
| 想把读到的设计沉淀下来 | 4A（ADR）/ 4B（方案卡）/ 4C（原子卡） |
| 想确认一张卡能不能入库 | 场景五（审查）—— 真正的瓶颈，绝不能跳 |
| 入库前整理格式 | 场景六（入库） |
| 维护总目录 | 场景七（目录维护） |
| AI 总结太浅，要追问 | 场景十三（深挖） |
| 读完多个项目找共性 | 场景十二（跨项目归纳） |
| 设计新系统前查经验 | 场景八（检索）—— 经验 ≥ 20 条后再启用 |
| 真正动手写代码 | 场景九（落地计划）+ 场景十（小步执行） |

**暂时用不上的**：场景十一（经验过期审查）——库还很年轻，等 6-12 个月后再启用。

## 4. 节奏与检验标准

### 节奏建议

- **不要试图一周读完整个项目**。gbrain 这种规模（100 ops + 84 CLI_ONLY + 41 个 core 子目录），全读完至少 2-3 个月。
- **每个模块 2-3 天**：1 天深读（场景二/三）、1 天提炼（4C）、半天审查（场景五）+ 入库。
- **一次只开一条线**。别同时开 hybrid search 和 minions 两条线，上下文切换成本高。
- **每周回看一次总目录**。看自己这周加了什么、和已有的有没有冲突或可合并。

### 检验标准（学没学到位的自检）

学完一个模块，你应该能**不查源码**回答下面 5 个问题中的至少 4 个：

1. 这个模块的核心抽象是什么？（一句话，不含形容词）
2. 它的主流程经过哪几个函数？（按顺序说出函数名和文件路径）
3. 它的关键阈值/魔法数字有哪些？为什么是这个值？（精确到数字）
4. 它在什么分支下会失败？失败时回退、跳过还是中断？（每条分支都要有去向）
5. 如果让我在新项目里复刻这个设计，第一步要做什么？

答不上来——说明还停在 4B 方案卡的概括层，回去往 4C 原子卡切。

## 5. 常见误区与对策

| 误区 | 表现 | 对策 |
|---|---|---|
| **经验卡一生成就入库** | draft 直接变 reviewed 没过审查 | 守场景五；宁可少入库 |
| **停在方案卡不切原子卡** | 卡里说"用 RRF 融合"但不给 score 公式 | 觉得空就往 4C 切 |
| **路径函数名脑补** | 附录写 `resolveRequestedSourceScope` 实际是 `resolveRequestedScope` | 每个引用都要打开源码核一遍 |
| **把经验库当读书笔记** | 卡里大段复述源码、贴原项目专有名词 | 正文必须脱离原项目，证据只进附录 |
| **一次读太多** | "今天读完整个 search 目录" | 一个模块一个流程地切，模板场景二/三的范围就是边界 |
| **跳过模块深读直接流程深读** | 流程深读时找不到对象锚点，变成"看一行忘一行" | 先场景二建对象地图，再场景三追调用链 |
| **学完不复用** | 经验入库就完事，从不在自己项目里试 | 阶段 3 的跨项目归纳 + 真正在新项目里用一次 |

## 附录：本指南与模板的关系

本指南**不替代模板**——模板是 13 个场景的 prompt 集合，本指南是"在学一个项目时，按什么顺序、用什么节奏调用这些 prompt"的元方法论。

两者分工：

- **模板**：每次提炼经验时，复制对应场景的 prompt 贴给 AI
- **本指南**：决定"现在该用模板的哪个场景、用完产出放哪、下一步做什么"

如果你完全按本指南走，一个新项目从 clone 到产出第一批 reviewed 经验，大概 1-2 周。之后每多读一个模块，加 2-3 天。读到 20 条经验后，就可以开始用场景八（检索）反哺自己的开发了。
