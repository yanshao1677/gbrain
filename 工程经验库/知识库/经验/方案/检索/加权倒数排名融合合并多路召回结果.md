# 加权倒数排名融合合并多路召回结果

## 1. 问题

检索系统同时跑多路召回（向量、关键词、关系图谱、跨模态图像），每路返回一个独立排序列表。要把它们合成一个最终排序，且满足：

- 不能依赖各路原始分数（向量 cosine 与 BM25 分数量纲不同，直接加权无意义）
- 一路排名好不应让该路垄断顶部（防 vector-only 漂移）
- 同一文档在多路出现应被奖励（共识信号）
- 不同查询意图下，各路贡献权重应不同（实体查询偏关键词、事件查询偏稀有词）

不解决会怎样：直接用任一路的分数当最终分 → 该路的盲区变系统盲区；统一权重 → 实体查询和事件查询用同一组权重，召回质量在意图边界处塌陷。

## 2. 适用约束

- 至少 2 路独立召回，每路返回有意义的排名（非随机）
- 各路分数不可比或量纲不一
- 召回列表长度有限（典型 ≤100，RRF 不处理无限流）
- 存在分类器能给出查询意图（无分类器则退化为等权 RRF，仍可用本方案的阶段 1）
- 跨进程无状态需求（融合纯函数，无外部依赖）

## 3. 核心思路

**用每路排名的倒数 `1/(k+rank)` 作为该路对该文档的投票，跨路累加；意图分类通过调整每路的 `k`（`k_eff = k_base / weight`）来倾斜——weight>1 → k 变小 → 该路 top 排名贡献更强，而非简单乘分数。**

关键：加权发生在"排名"维度而非"分数"维度。weight 改变的是 top 排名的陡峭度，不是线性放大分数——这保住了 RRF 的"共识优先"语义。

## 4. 通用结构

```
召回阶段          每路独立 k（来自意图分类）
─────────────────────────────────────────────
vector list   ──── k = k_base / vectorWeight ──┐
keyword list  ──── k = k_base / keywordWeight ─┤
relational    ──── k = k_base（无意图加权）    ├─→ rrfFusionWeighted
image list    ──── k = k_base / imageWeight ───┘        │
                                                         ▼
                                              Map<dedupKey, {result, score}>
                                                  score = Σ 1/(k_i + rank_i)
                                                         │
                                                         ▼
                                              归一化（÷ maxScore）→ 0-1
                                                         │
                                                         ▼
                                              source-type boost（可选）
                                                         │
                                                         ▼
                                              按 boosted score 降序
```

数据结构：
- `Map<string, { result: SearchResult; score: number }>` —— key 是融合去重键
- 融合去重键 = `source_id:slug:chunk_id`（chunk_id 为 null 时 fallback `chunk_text.slice(0, 50)`）—— 三段式保证跨 source、跨同 slug 不同 chunk 不误合并

## 5. 处理流程

1. 意图分类器产出 `intent ∈ {entity, temporal, event, general}`（确定性，无 LLM）
2. 查 `weightsForIntent(intent)` 表得到 `{keywordWeight, vectorWeight, ...}`
3. 每路召回产出 `SearchResult[]`
4. 对每路计算 `k_eff = effectiveRrfK(k_base, weight) = k_base / weight`（weight≤0 时回退 k_base）
5. `rrfFusionWeighted`：遍历每路每个结果，`rrfScore = 1/(k_eff + rank)`（rank 从 0 起），按 dedupKey 累加进 Map
6. 全部累加完，`maxScore = max(所有 score)`，每条 `score /= maxScore` 归一化到 0-1
7. 若 `applyBoost` 且 `chunk_source === 'compiled_truth'`，`score *= 2.0`
8. 按 `score` 降序输出（score 相同时未指定 tiebreaker，见阶段 1 原子卡"失败处理"）

## 6. 异常处理

- 某路召回为空列表：该路贡献 0，不影响其他路；不抛错
- 全部路为空：返回 `[]`，不抛错
- `weight ≤ 0`：`effectiveRrfK` 回退到 `k_base`（防御性，正常分类器不产生 ≤0 的 weight）
- `maxScore === 0`（所有 rrfScore 都为 0，理论不可能）：跳过归一化与 boost，直接输出原始 score（全 0）的排序
- dedupKey 冲突（同 key 不同结果对象）：第一次见到的 result 入 Map，后续只累加 score——**result 对象不被替换**，这意味着"哪一路的 result 元数据被保留"取决于路传入顺序，调用方需保证顺序稳定

## 7. 具体语言实现（TypeScript 骨架）

```typescript
const RRF_K_BASE = 60;
const COMPILED_TRUTH_BOOST = 2.0;

interface SearchResult {
  slug: string;
  chunk_id: string | null;
  chunk_text: string;
  source_id?: string;
  chunk_source?: string;
  score: number;
}

interface IntentWeights {
  keywordWeight: number;
  vectorWeight: number;
  // ...
}

function rrfKey(r: SearchResult): string {
  const source = r.source_id ?? 'default';
  return `${source}:${r.slug}:${r.chunk_id ?? r.chunk_text.slice(0, 50)}`;
}

function effectiveRrfK(baseK: number, weight: number): number {
  if (weight <= 0) return baseK;
  return baseK / weight;
}

export function rrfFusionWeighted(
  lists: Array<{ list: SearchResult[]; k: number }>,
  applyBoost = true,
): SearchResult[] {
  const scores = new Map<string, { result: SearchResult; score: number }>();

  for (const { list, k } of lists) {
    for (let rank = 0; rank < list.length; rank++) {
      const r = list[rank];
      const key = rrfKey(r);
      const rrfScore = 1 / (k + rank);   // rank 从 0 起
      const existing = scores.get(key);
      if (existing) {
        existing.score += rrfScore;
      } else {
        scores.set(key, { result: r, score: rrfScore });
      }
    }
  }

  const entries = Array.from(scores.values());
  if (entries.length === 0) return [];

  const maxScore = Math.max(...entries.map(e => e.score));
  if (maxScore > 0) {
    for (const e of entries) {
      e.score = e.score / maxScore;       // 归一化到 0-1
      const boost = applyBoost && e.result.chunk_source === 'compiled_truth'
        ? COMPILED_TRUTH_BOOST : 1.0;
      e.score *= boost;
    }
  }

  return entries
    .sort((a, b) => b.score - a.score)
    .map(({ result, score }) => ({ ...result, score }));
}
```

## 8. 测试点

- 两路同一文档出现 → score 是单路的两倍（共识奖励）
- rank=0 的文档 score = 1/k；rank=k 的文档 score = 1/(2k)
- weight=1.20 的路，其 rank=0 文档 score = 1/(k/1.20) = 1.20/k > 1/k（倾斜生效）
- weight≤0 → 回退 k_base（不抛错）
- 同 dedupKey 跨路累加，不替换 result 对象
- `compiled_truth` 归一化后 ×2.0；非 compiled_truth ×1.0
- `applyBoost=false`（temporal/event 意图）跳过 boost
- 空列表路不影响其他路
- 全空 → 返回 `[]`

## 9. 适用场景 / 不适用场景

**适用**：多路召回融合、各路分数不可比、需要共识信号、有意图分类器
**不适用**：
- 单路召回（直接用该路分数）
- 各路分数已归一化且量纲一致（可直接加权平均，RRF 反而丢分数信息）
- 召回列表无排名意义（随机序）—— RRF 假设 rank 有信号
- 强需分数可解释性（RRF score 是合成的，不对应任何物理量）

## 10. 风险与反模式

- **weight 直接乘分数而非调 k** —— 破坏 RRF 共识语义，变回加权平均
- **dedupKey 只用 slug** —— 同 slug 不同 chunk 被合并，丢细粒度
- **归一化前 boost** —— boost 被随后的归一化抹掉
- **k_base 取太小（如 5）** —— top-1 与 top-10 的 score 差过大，共识信号被压制
- **k_base 取太大（如 1000）** —— 排名差异被抹平，退化为计数
- **路传入顺序不稳定** —— 同 dedupKey 的 result 元数据保留取决于顺序，调用方需固定顺序
