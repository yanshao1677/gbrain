# 阶段2：意图感知的每路有效 k 调整

## 触发条件

`hybridSearch` 主流程已通过意图分类器（`classifyQuery`）得到 `intent ∈ {entity, temporal, event, general}`，即将调用 `rrfFusionWeighted`；需要把意图翻译成"每路独立 k"，让某些意图在某一路上贡献更强。

## 输入字段

| 字段 | 类型 | 来源 | 可选 |
|------|------|------|------|
| `intent` | `'entity' \| 'temporal' \| 'event' \| 'general'` | `classifyQuery(query)` 确定性分类，无 LLM | 否 |
| `baseRrfK` | `number` | `opts.rrfK ?? RRF_K`（默认 60） | 是，默认 60 |
| `keywordWeight` | `number` | `weightsForIntent(intent).keywordWeight` | 否 |
| `vectorWeight` | `number` | `weightsForIntent(intent).vectorWeight` | 否 |
| `cross_modal_both_text_weight` | `number` | resolved mode bundle（跨模态 'both' 模式） | 是，仅 'both' 模式用 |
| `cross_modal_both_image_weight` | `number` | resolved mode bundle | 是，仅 'both' 模式用 |

## 判定规则

1. **查表得 weight**（`weightsForIntent`，源码 `intent-weights.ts:64-91`）：

   | intent | keywordWeight | vectorWeight | suggestedRecency | exactMatchBoost |
   |--------|---------------|--------------|------------------|-----------------|
   | `entity` | **1.15** | 1.0 | `null` | 1.25 |
   | `temporal` | 1.0 | 1.0 | `'on'` | 1.0 |
   | `event` | **1.20** | **0.95** | `'on'` | 1.10 |
   | `general` | 1.0 | 1.0 | `null` | 1.0 |

   - 未知 intent → 回退 `DEFAULT_WEIGHTS`（即 general）
   - 表是硬编码常量，非配置项

2. **计算每路有效 k**（`effectiveRrfK`，源码 `intent-weights.ts:109-112`）：
   ```
   effectiveRrfK(baseK, weight):
     if weight <= 0: return baseK        // 防御性回退
     return baseK / weight
   ```
   - `weight > 1` → `k_eff < k_base` → 该路 top 排名贡献**更强**（1/(k_eff+0) > 1/(k_base+0)）
   - `weight < 1` → `k_eff > k_base` → 该路贡献**更平**（top 被压制，长尾被抬高）
   - `weight = 1` → `k_eff = k_base`（no-op）

3. **每路 k 的具体计算**（`hybrid.ts:1327-1337`）：
   ```
   baseRrfK = opts.rrfK ?? 60
   keywordK = effectiveRrfK(baseRrfK, intentWeights.keywordWeight)
   vectorK  = effectiveRrfK(baseRrfK, intentWeights.vectorWeight)
   textRrfK = effectiveRrfK(baseRrfK, resolvedMode.cross_modal_both_text_weight)    // 仅 'both' 模式
   imageRrfK = effectiveRrfK(baseRrfK, resolvedMode.cross_modal_both_image_weight)  // 仅 'both' 模式
   ```

4. **relational 路不参与意图加权**（`hybrid.ts:1359-1361`）：用 `baseRrfK`，不调 `effectiveRrfK`。注释称"neutral weight: competes evenly with keyword/vector, not dominating"。

5. **各 intent 的实际 k 值表**（k_base=60）：

   | intent | keywordK | vectorK | 说明 |
   |--------|----------|---------|------|
   | entity | 60/1.15 ≈ **52.17** | 60/1.0 = 60 | 关键词路 top 更陡 |
   | temporal | 60 | 60 | 等权，靠 recency 而非 k 调整 |
   | event | 60/1.20 = **50** | 60/0.95 ≈ **63.16** | 关键词陡、向量平 |
   | general | 60 | 60 | 等权 |

**魔法数字追问**：
- `entity.keywordWeight = 1.15`、`event.keywordWeight = 1.20`：源码注释（`intent-weights.ts:66-72, 82-85`）解释了方向（entity 查询用户知道名字，reward 精确匹配；event 查询有稀有命名实体，关键词搜得准），但 **1.15 与 1.20 的具体数值依据未说明**——推测是经验调参。
- `event.vectorWeight = 0.95`：注释说"vector smears across paraphrases"，但 **0.95 而非 0.9 或 0.8 的依据未说明**。
- `temporal` 的 k 不调整（全 1.0）：注释说"recency tilt is the whole game"，temporal 靠 `suggestedRecency: 'on'` 而非 k 调整——**这是设计决策，有注释解释**。

## 状态读写位置

- **读**：`INTENT_WEIGHTS` 常量表（模块级，进程内只读）
- **写**：无；纯函数返回新 `IntentWeights` 对象
- **跨进程语义**：无需变化——纯查表 + 算术

## 正常路径

1. `classifyQuery(query)` → `intent`（确定性，无 LLM）
2. `intentWeights = weightsForIntent(intent)`（查表，未知 intent 回退 DEFAULT_WEIGHTS）
3. `baseRrfK = opts.rrfK ?? RRF_K`（默认 60）
4. 每路算 `k_eff = effectiveRrfK(baseRrfK, weight)`：
   - keyword 路 → `keywordK`
   - vector 路 → `vectorK`（'both' 模式拆成 textRrfK + imageRrfK）
   - relational 路 → 直接用 `baseRrfK`（不调 effectiveRrfK）
5. 组装 `allLists: Array<{list, k}>`，传给 `rrfFusionWeighted`

**先后顺序约束**：
- 必须先分类再算 k（k 依赖 intent）
- 必须在 `rrfFusionWeighted` 调用前算完所有路的 k

## 分支路径

| 条件 | 行为 |
|------|------|
| intent = `general` | 所有 weight = 1.0，k_eff = k_base，退化为等权 RRF |
| intent = `temporal` | k 不调整，但 `suggestedRecency: 'on'` 会在后续 post-fusion 阶段启用 recency boost |
| intent = `entity` / `event` | keywordK < k_base（关键词路 top 更陡）；event 还会让 vectorK > k_base（向量路 top 被压） |
| 未知 intent（分类器返回意外值） | `weightsForIntent` 回退 `DEFAULT_WEIGHTS`，等同 general |
| `weight ≤ 0` | `effectiveRrfK` 回退 `baseK`（防御性，正常不发生） |
| `opts.rrfK` 显式传入 | 用传入值作 baseK，而非默认 60（per-call override） |
| 跨模态 'both' 模式 | vector 路拆成 text + image 两路，各用 `cross_modal_both_text_weight` / `cross_modal_both_image_weight` 算 k |
| relational 路存在 | 用 `baseRrfK`，不参与 intent 加权（neutral） |

## 失败处理

- **分类器无异常路径**：`classifyQuery` 是确定性函数，不抛错；最坏返回 `'general'`
- **weight 为 NaN/Infinity**：`effectiveRrfK` 的 `weight <= 0` 检查不捕获 NaN（NaN <= 0 是 false），会算出 NaN k → `1/(NaN+rank) = NaN` → 整个 score 变 NaN → 排序未定义。**证据不足：源码未见 NaN 防御，推测分类器保证不返回 NaN，但需在 classifyQuery 实现中确认**
- **k_base 为 0 或负**：`1/(0+rank)` rank=0 时除零 → Infinity。**证据不足：源码未见 k_base 合法性校验，依赖调用方传正数**

## 幂等性 / 一致性约束

- **确定性**：相同 intent + 相同 baseRrfK → 相同 k_eff（纯算术）
- **无副作用**：不修改 intentWeights 表，不修改 SearchResult
- **对后续阶段约束**：
  - k_eff 必须为正数（否则阶段 1 的 `1/(k+rank)` 除零）
  - 不同路的 k_eff 相互独立，不互相影响
  - relational 路用 baseRrfK 是**有意的不对称**——后续阶段不可"为了对称"也给 relational 加 intent weight，那会让关系召回在 entity 查询里压过 keyword

## 代码骨架

```typescript
type QueryIntent = 'entity' | 'temporal' | 'event' | 'general';

interface IntentWeights {
  keywordWeight: number;
  vectorWeight: number;
  suggestedRecency: 'off' | 'on' | 'strong' | null;
  exactMatchBoost: number;
}

const DEFAULT_WEIGHTS: IntentWeights = {
  keywordWeight: 1.0,
  vectorWeight: 1.0,
  suggestedRecency: null,
  exactMatchBoost: 1.0,
};

const INTENT_WEIGHTS: Record<QueryIntent, IntentWeights> = {
  entity:   { keywordWeight: 1.15, vectorWeight: 1.0,  suggestedRecency: null,  exactMatchBoost: 1.25 },
  temporal: { keywordWeight: 1.0,  vectorWeight: 1.0,  suggestedRecency: 'on',  exactMatchBoost: 1.0 },
  event:    { keywordWeight: 1.20, vectorWeight: 0.95, suggestedRecency: 'on',  exactMatchBoost: 1.10 },
  general:  DEFAULT_WEIGHTS,
};

export function weightsForIntent(intent: QueryIntent): IntentWeights {
  return INTENT_WEIGHTS[intent] ?? DEFAULT_WEIGHTS;
}

export function effectiveRrfK(baseK: number, weight: number): number {
  if (weight <= 0) return baseK;
  return baseK / weight;
}

// 调用点（hybrid.ts:1327-1361 简化）
function buildListsForFusion(
  intent: QueryIntent,
  baseRrfK: number,
  keywordResults: SearchResult[],
  vectorLists: SearchResult[][],
  relationalList: SearchResult[],
): Array<{ list: SearchResult[]; k: number }> {
  const w = weightsForIntent(intent);
  const keywordK = effectiveRrfK(baseRrfK, w.keywordWeight);
  const vectorK = effectiveRrfK(baseRrfK, w.vectorWeight);

  const allLists: Array<{ list: SearchResult[]; k: number }> = [
    ...vectorLists.map(list => ({ list, k: vectorK })),
    { list: keywordResults, k: keywordK },
  ];
  // relational 路用 baseRrfK，不参与 intent 加权
  if (relationalList.length > 0) {
    allLists.push({ list: relationalList, k: baseRrfK });
  }
  return allLists;
}
```

## 最小验证清单

- intent=entity, baseRrfK=60 → keywordK ≈ 52.17, vectorK = 60
- intent=event, baseRrfK=60 → keywordK = 50, vectorK ≈ 63.16
- intent=general → keywordK = vectorK = 60（等权）
- intent=temporal → keywordK = vectorK = 60（k 不调整，靠 recency）
- 未知 intent（如 `'foo'`） → 回退 DEFAULT_WEIGHTS，k = 60
- weight=0 → effectiveRrfK 回退 baseK（不抛错）
- weight=1.0 → k_eff = baseK（no-op）
- weight=1.20, baseK=60 → k_eff = 50（1/(50+0) = 0.02 > 1/(60+0) ≈ 0.0167，top 贡献更强）
- relational 路始终用 baseRrfK，不受 intent 影响
- `opts.rrfK=100` 传入 → 所有路的 baseK 用 100，intent 调整在此基础上算
