# 命名能力包与版本化缓存指纹防污染

## 1. 问题

检索系统有 10+ 可调旋钮（结果 limit、token budget、LLM expansion、reranker、graph signals、embedding 列、schema pack…）。若语义缓存仅键 `(tenant, query_text)`：

- **conservative** 写入的小结果集被 **tokenmax** 读回 → 召回不足
- 换 embedding 列维度后旧向量仍命中 → 错误相似度
- schema pack 切换后 type closure 变化 → 答非所问

## 2. 适用约束

- 存在 named preset（安装时选一次）+ per-call override + config key override 三层
- 有 query-result 语义缓存（embedding 相似度 + TTL）
- 旋钮组合影响**结果集 identity**，不仅是 latency
- 下游 agent 成本对 payload 大小敏感（mode 间可达 25x 级差）

## 3. 核心思路

1. **ModeBundle**：每个命名 mode（如 conservative / balanced / tokenmax）冻结一份完整 knob 默认值表  
2. **Resolution chain**：`per-call opts → config keys → MODE_BUNDLES[mode] → fallback bundle`  
3. **knobsHash**：对**解析后的最终 knob 集**做固定顺序串联 + SHA-256，前缀 `KNOBS_HASH_VERSION`；缓存读写必须 `WHERE knobs_hash = $hash`  
4. 任何新增影响结果的 knob → bump version → 旧行自然 miss（一次性冷启动）

## 4. 通用结构

```
resolveSearchMode(opts, config) → ResolvedSearchKnobs
knobsHash(knobs, { embeddingColumn, embeddingModel, schemaPack }) → string
query_cache: (source_scope, query_embedding, knobs_hash) → results JSON
```

ModeBundle 字段示例（每 mode 全部必填，便于 diff）：

| Knob | conservative | balanced | tokenmax |
|------|-------------|----------|----------|
| tokenBudget | 4000 | 12000 | undefined（off） |
| expansion | false | false | true |
| searchLimit | 10 | 25 | 50 |
| reranker_enabled | false | true | true |
| graph_signals | false | true | true |
| relationalRetrieval | false | true | true |

（cache：enabled=true, similarity_threshold=0.92, ttl_seconds=3600 三 mode 相同）

## 5. 处理流程

1. `hybridSearch` 入口调用 `resolveSearchMode`（**非**仅 cached wrapper——eval 与生产同路径）
2. 合并 per-call / config / bundle 得 `ResolvedSearchKnobs`
3. 计算 `hash = knobsHash(knobs, ctx)`，`ctx` 含 embedding 列、provider、schema pack id/version
4. 缓存 lookup：`source_id` + embedding similarity < threshold + **`knobs_hash = hash`**
5. Miss → 全 pipeline → INSERT 行带 `knobs_hash`
6. NULL `knobs_hash` 历史行：lookup 排除，不 serve

## 6. 异常处理

- Reranker 超时/无 API key：**fail-open** 保持 RRF 序，audit JSONL；不污染 cache key
- 未知 mode + remote caller：**reject**（防 expensive bundle 升级）
- 未知 mode + local：**loud reject**（operator 可见）

## 7. 具体语言实现（TypeScript 骨架）

```typescript
const KNOBS_HASH_VERSION = 11; // bump on any result-affecting knob

const MODE_BUNDLES = {
  conservative: { tokenBudget: 4000, expansion: false, searchLimit: 10, /* ... */ },
  balanced:     { tokenBudget: 12000, expansion: false, searchLimit: 25, reranker_enabled: true, /* ... */ },
  tokenmax:     { tokenBudget: undefined, expansion: true, searchLimit: 50, /* ... */ },
} as const;

function resolveSearchMode(perCall: Partial<Knobs>, config: Config): ResolvedKnobs {
  const mode = perCall.mode ?? config.search?.mode ?? 'balanced';
  const bundle = MODE_BUNDLES[mode] ?? MODE_BUNDLES.balanced;
  return { ...bundle, ...pickConfigOverrides(config), ...perCall, resolved_mode: mode };
}

function knobsHash(knobs: ResolvedKnobs, ctx?: KnobsHashContext): string {
  const parts = [
    `v=${KNOBS_HASH_VERSION}`,
    `mode=${knobs.resolved_mode}`,
    `limit=${knobs.searchLimit}`,
    `expansion=${knobs.expansion}`,
    `tokenBudget=${knobs.tokenBudget ?? 'off'}`,
    `rerank=${knobs.reranker_enabled}`,
    `relational=${knobs.relationalRetrieval}`,
    `embCol=${ctx?.embeddingColumn ?? 'embedding'}`,
    `embModel=${ctx?.embeddingModel ?? 'default'}`,
    `pack=${ctx?.schemaPack ?? 'none'}@${ctx?.schemaPackVersion ?? '0'}`,
    // ... fixed-order list; never reorder without version bump
  ];
  return createHash('sha256').update(parts.join('|')).digest('hex');
}

// cache lookup
// WHERE source_id = $1 AND knobs_hash = $4 AND embedding <=> $q < $threshold
```

## 8. 测试点

- conservative query 写 cache → tokenmax 同 query miss（不同 hash）
- bump `KNOBS_HASH_VERSION` → 全库旧行 miss
- 换 embedding column → hash 变 → miss
- 同 mode 同 knobs 第二次 hit（相似度 + hash 均 match）
- eval replay 走 bare `hybridSearch` 仍 respect mode（与 production parity）

## 9. 适用场景 / 不适用场景

**适用**：多 preset 检索、语义缓存、embedding/schema 可切换  
**不适用**：无缓存、旋钮少且全局固定、结果不随 config 变——简单 key 即可

## 10. 风险与反模式

- 只在 cached wrapper 做 mode resolve → eval/production 分叉
- hash 漏 fold 关键 knob →  silent contamination（需 version bump 纪律）
- per-call override 不进 hash → 污染
- mode 过多无命名 bundle → 运营不可理解；应 ≤3–5 个命名 tier + 文档化成本矩阵
